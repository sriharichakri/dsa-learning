AVL Tree

Phase 9 Hard

📖 What is an AVL Tree?

An AVL Tree (named after inventors Adelson-Velsky and Landis) is a self-balancing Binary Search Tree where the difference in heights between the left and right subtrees of any node (called the balance factor) is at most 1.

When an insertion or deletion violates this balance, the tree performs rotations to restore balance, guaranteeing O(log n) time for all operations — even in the worst case.

💡 Key Insight

A regular BST can degenerate into a linked list (O(n) operations) if elements are inserted in sorted order. AVL trees prevent this by automatically rebalancing after every insertion/deletion. The balance factor = height(left) − height(right) must always be in {-1, 0, 1}.

🌍 Real-Life Analogy

📚 Self-Adjusting Bookshelf

Imagine a bookshelf that automatically rearranges itself whenever one side gets too heavy. If you keep stacking books on the left, the shelf detects the imbalance and shifts some books to the right to stay balanced. Each "rotation" is like the shelf sliding a group of books to keep everything evenly distributed — ensuring you can always reach any book quickly.

  • Seesaw: If one side is heavier, move the pivot or shift weight to balance it.
  • Balanced diet: Too much of one thing? Adjust by adding variety.
  • Team management: If one team grows too large, reorganize by splitting responsibilities.

🔧 AVL Tree Implementation

JavaScript — AVL Tree
class AVLNode {
    constructor(val) {
        this.val = val;
        this.left = null;
        this.right = null;
        this.height = 1; // New node is a leaf
    }
}

class AVLTree {
    constructor() {
        this.root = null;
    }

    getHeight(node) {
        return node ? node.height : 0;
    }

    getBalanceFactor(node) {
        return node ? this.getHeight(node.left) - this.getHeight(node.right) : 0;
    }

    updateHeight(node) {
        node.height = 1 + Math.max(this.getHeight(node.left), this.getHeight(node.right));
    }

    // Right Rotation (LL case)
    rightRotate(y) {
        const x = y.left;
        const T2 = x.right;

        x.right = y;
        y.left = T2;

        this.updateHeight(y);
        this.updateHeight(x);
        return x; // New root
    }

    // Left Rotation (RR case)
    leftRotate(x) {
        const y = x.right;
        const T2 = y.left;

        y.left = x;
        x.right = T2;

        this.updateHeight(x);
        this.updateHeight(y);
        return y; // New root
    }

    // Balance the node after insertion/deletion
    balance(node, val) {
        const bf = this.getBalanceFactor(node);

        // LL Case
        if (bf > 1 && val < node.left.val) {
            return this.rightRotate(node);
        }
        // RR Case
        if (bf < -1 && val > node.right.val) {
            return this.leftRotate(node);
        }
        // LR Case
        if (bf > 1 && val > node.left.val) {
            node.left = this.leftRotate(node.left);
            return this.rightRotate(node);
        }
        // RL Case
        if (bf < -1 && val < node.right.val) {
            node.right = this.rightRotate(node.right);
            return this.leftRotate(node);
        }
        return node; // Already balanced
    }

    insert(val) {
        this.root = this._insert(this.root, val);
    }

    _insert(node, val) {
        // Standard BST insert
        if (!node) return new AVLNode(val);

        if (val < node.val) node.left = this._insert(node.left, val);
        else if (val > node.val) node.right = this._insert(node.right, val);
        else return node; // No duplicates

        this.updateHeight(node);
        return this.balance(node, val);
    }
}

Each rotation is O(1). At most O(log n) rotations per insert/delete (one rotation per ancestor on the path).

⚠️ Common Mistakes

❌ Forgetting to update heights after rotation

After a rotation, both the old root and new root have changed subtrees. Always update heights of the lower node first, then the higher node.

❌ Confusing rotation direction

LL imbalance → Right rotation (counterintuitive!). Think: "the problem is on the Left-Left, so rotate Right to fix it."

❌ Not handling LR/RL as double rotations

LR and RL require TWO rotations. First convert to LL or RR, then do a single rotation. Missing the first rotation leads to incorrect trees.

  • AVL trees are rarely coded from scratch in interviews — but understanding why self-balancing matters is crucial.
  • Know: balance factor, the 4 rotation types, and when each applies.
  • Compare with Red-Black Trees: AVL is more strictly balanced (faster lookups) but requires more rotations on insert/delete.
  • Common question: "How would you check if a binary tree is balanced?" → LC 110: Balanced Binary Tree.
  • AVL guarantees height ≤ 1.44 × log₂(n), so operations are truly O(log n).

📝 Quick Revision Notes

  • AVL = self-balancing BST. Balance factor = height(left) − height(right) ∈ {-1, 0, 1}.
  • 4 rotations: LL→Right, RR→Left, LR→Left+Right, RL→Right+Left.
  • All operations guaranteed O(log n) — no degenerate cases.
  • Update height bottom-up after every insert/delete.
  • More balanced than Red-Black trees → faster search, slower insert/delete.

🃏 Flash Cards

What is the balance factor of an AVL tree node?
Balance Factor = height(left subtree) − height(right subtree). It must be -1, 0, or 1 for every node in the tree.
When do you perform a Left-Right (LR) rotation?
When the balance factor is > 1 (left-heavy) AND the new node was inserted into the right subtree of the left child. First left-rotate the left child, then right-rotate the node.
How does AVL differ from Red-Black Tree?
AVL is more strictly balanced (max height difference = 1) → faster lookups. Red-Black allows more imbalance → fewer rotations on insert/delete. Java's TreeMap uses Red-Black; databases often use AVL.
