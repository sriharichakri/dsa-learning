Aho-Corasick Algorithm

Phase 15 Hard

📖 What is the Aho-Corasick Algorithm?

Aho-Corasick is a multi-pattern string matching algorithm that finds all occurrences of multiple patterns in a text simultaneously in O(n + m + z) time, where n is the text length, m is the total length of all patterns, and z is the number of matches. It combines a Trie with failure links (similar to KMP's failure function) to create an automaton.

While KMP handles one pattern at a time, Aho-Corasick handles thousands of patterns in a single pass through the text — making it ideal for applications like virus scanning, spam filtering, and DNA sequence analysis.

🕵️ Real-Life Analogy: Airport Security Watchlist

Imagine airport security checking every passenger name against a watchlist of thousands of names. Instead of scanning the list for each name separately, they build a smart lookup structure. As each character of a passenger name is read, the system simultaneously checks all possible watchlist matches. If a partial match fails, it jumps to the longest alternative partial match (failure link) rather than starting over. Aho-Corasick is this smart lookup system!

⚙️ How Aho-Corasick Works

Aho-Corasick Automaton for patterns: ["he", "she", "his", "hers"]
Step 1: Build Trie root / | \ h s (other chars → root) / \ \ e i h | | | r s e | s Step 2: Add Failure Links (BFS) Fail link of each node = longest proper suffix that is also a prefix of some pattern in the trie. 'h' → root (no suffix "h" starts another pattern... well "h" does but it's self) 's' → root "he" → root (suffix "e" not in trie as prefix) "sh" → 'h' (suffix "h" is a prefix in trie!) "she" → "he" (suffix "he" is in the trie!) ↑ This means when we match "she", we also match "he"! Step 3: Search text "ahishers" a → root (no match) h → node 'h' i → node 'hi' s → node 'his' → OUTPUT "his" at pos 1 fail → 's' (continue) h → node 'sh' e → node 'she' → OUTPUT "she" at pos 3 fail link → 'he' → OUTPUT "he" at pos 4 r → node 'her' s → node 'hers' → OUTPUT "hers" at pos 4

🔧 Implementation

JavaScript — Aho-Corasick Multi-Pattern Matching
class AhoCorasick {
    constructor() {
        this.goto = [{}];       // Trie transitions: goto[state][char] = nextState
        this.fail = [0];        // Failure links
        this.output = [[]];     // Output: which patterns end at this state
        this.stateCount = 1;
    }

    // Add a pattern to the trie
    addPattern(pattern, id) {
        let state = 0;
        for (const ch of pattern) {
            if (!(ch in this.goto[state])) {
                this.goto[state][ch] = this.stateCount;
                this.goto.push({});
                this.fail.push(0);
                this.output.push([]);
                this.stateCount++;
            }
            state = this.goto[state][ch];
        }
        this.output[state].push({ id, pattern });
    }

    // Build failure links using BFS
    build() {
        const queue = [];

        // Depth-1 nodes: fail links point to root
        for (const ch in this.goto[0]) {
            const s = this.goto[0][ch];
            this.fail[s] = 0;
            queue.push(s);
        }

        while (queue.length > 0) {
            const u = queue.shift();
            for (const ch in this.goto[u]) {
                const v = this.goto[u][ch];
                queue.push(v);

                // Find failure state
                let f = this.fail[u];
                while (f !== 0 && !(ch in this.goto[f])) {
                    f = this.fail[f];
                }
                this.fail[v] = ch in this.goto[f] ? this.goto[f][ch] : 0;

                // Merge output with failure link's output
                this.output[v] = this.output[v].concat(this.output[this.fail[v]]);
            }
        }
    }

    // Search text for all pattern occurrences
    search(text) {
        const results = [];
        let state = 0;

        for (let i = 0; i < text.length; i++) {
            const ch = text[i];

            // Follow failure links until we find a transition or reach root
            while (state !== 0 && !(ch in this.goto[state])) {
                state = this.fail[state];
            }
            state = ch in this.goto[state] ? this.goto[state][ch] : 0;

            // Record all matches at this state
            for (const match of this.output[state]) {
                results.push({
                    pattern: match.pattern,
                    position: i - match.pattern.length + 1
                });
            }
        }
        return results;
    }
}

// Usage
const ac = new AhoCorasick();
ac.addPattern("he", 0);
ac.addPattern("she", 1);
ac.addPattern("his", 2);
ac.addPattern("hers", 3);
ac.build();

console.log(ac.search("ahishers"));
// [{pattern:"his", position:1}, {pattern:"she", position:3},
//  {pattern:"he", position:4}, {pattern:"hers", position:4}]

⚠️ Common Mistakes

❌ Forgetting to merge output along failure links

A node's output should include not just patterns ending there, but also all patterns at its failure link chain. Without merging, shorter patterns embedded in longer ones are missed (e.g., "he" inside "she").

❌ Not calling build() before search()

Failure links are computed by build() using BFS. Searching without building first means all failure links are 0 (root), and the automaton won't find overlapping patterns.

❌ Incorrect failure link for depth-1 nodes

All nodes at depth 1 (direct children of root) must have their failure links set to root. Some implementations miss this, causing infinite loops.

📝 Quick Revision Notes

  • Aho-Corasick = Trie + failure links (KMP-like) → multi-pattern automaton.
  • Build: insert all patterns into trie, then BFS to compute failure links.
  • Search: traverse automaton on text, follow failure links on mismatch, collect outputs.
  • Time: O(n + m + z). Space: O(m × Σ).
  • Output merging: each node's output = own patterns + failure link's output.

🃏 Flash Cards

What two data structures does Aho-Corasick combine?
1. Trie — stores all patterns and provides forward transitions (goto function). 2. Failure links (like KMP's failure function) — when a mismatch occurs, jump to the longest proper suffix of the current match that is also a prefix of some pattern. Together they form a finite automaton.
Why is output merging along failure links necessary?
A longer pattern's node might have a failure link to a shorter pattern's node. When we reach the longer pattern's node, we should also report the shorter pattern. Example: "she" node fails to "he" node — when "she" is matched, "he" is also matched at the same position. Without merging, "he" would be missed.
