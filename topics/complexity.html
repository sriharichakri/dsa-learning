Time & Space Complexity (Big O)

Phase 1 Beginner Friendly

📖 What is Complexity Analysis?

Imagine two delivery drivers both need to deliver 100 packages. Driver A delivers them one by one. Driver B sorts all packages by area first, then delivers them in one optimized route. Both finish the job — but Driver B is dramatically faster.

Complexity analysis is how we measure how fast an algorithm runs (time) and how much memory it uses (space) as the input grows larger.

💡 The Big Question

We don't measure in seconds (that depends on hardware). Instead we ask: "If I double the input size, how much longer does it take?"

🤔 Why Do We Need This?

Two solutions can both be "correct" but behave completely differently:

Input Size (n)O(n) AlgorithmO(n²) Algorithm
1010 ops100 ops
1,0001,000 ops1,000,000 ops
1,000,0001,000,000 ops1,000,000,000,000 ops 💀

At large inputs, an O(n²) solution can take hours while O(n) finishes in milliseconds. In interviews, choosing the right algorithm is what separates "accepted" from "time limit exceeded".

🌍 Real-Life Analogy

📚 Finding a Word in a Dictionary

O(n) — Linear: Start at page 1, read every word until you find it. For a 1000-page dictionary, you might check all 1000 pages.

O(log n) — Binary Search: Open the middle. Too early? Go to second half's middle. Too late? Go to first half's middle. For 1000 pages, you need at most ~10 checks!

O(1) — Direct Access: Someone tells you "page 547, word 3." You go straight there. No searching needed, regardless of dictionary size.

📊 Big O, Big Theta, Big Omega

  • Big O (O): Upper bound — "In the worst case, my algorithm takes at most this long." This is what interviews care about most.
  • Big Omega (Ω): Lower bound — "In the best case, it takes at least this long."
  • Big Theta (Θ): Tight bound — "It takes exactly this long (both upper and lower)."
💡 Interview Tip

When someone says "What's the time complexity?" they almost always mean Big O (worst case). But understanding best/average case helps you explain your solution better.

🔍 How to Calculate Big O

Rule 1: Drop Constants

// This runs 3n times → O(3n) → O(n)
for (let i = 0; i < 3 * n; i++) { ... }

// Constants don't matter for growth rate

Rule 2: Drop Lower-Order Terms

// O(n² + n + 1) → O(n²)
// As n grows, n² dominates everything else

Rule 3: Different Inputs = Different Variables

// Two different arrays → O(a + b), NOT O(n)
for (let x of arrA) { ... }  // O(a)
for (let y of arrB) { ... }  // O(b)

Rule 4: Nested Loops Multiply

// Outer: n times × Inner: n times = O(n × n) = O(n²)
for (let i = 0; i < n; i++) {
    for (let j = 0; j < n; j++) {
        // ...
    }
}

Rule 5: Sequential Code Adds

// O(n) + O(n) = O(2n) = O(n)
for (let i = 0; i < n; i++) { ... }  // O(n)
for (let j = 0; j < n; j++) { ... }  // O(n)

🧠 Space Complexity

Space complexity measures extra memory used by your algorithm (not counting the input itself).

// O(1) space — only using a few variables
function sum(arr) {
    let total = 0;  // one variable
    for (let x of arr) total += x;
    return total;
}

// O(n) space — creating a new array of size n
function doubled(arr) {
    let result = [];  // grows with input
    for (let x of arr) result.push(x * 2);
    return result;
}

// O(n) space — recursion uses call stack space!
function factorial(n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1); // n stack frames
}

⚠️ Common Mistakes

❌ Saying O(n) when it's actually O(n²)

Watch for hidden loops! Methods like .includes(), .indexOf(), .sort() inside a loop add complexity.

// ❌ This is O(n²), not O(n)!
for (let i = 0; i < arr.length; i++) {
    if (arr.includes(target - arr[i])) { ... }
    //    ^^^ includes is O(n) inside an O(n) loop!
}
❌ Forgetting recursion space

Each recursive call adds a frame to the call stack. A recursion of depth n uses O(n) space.

❌ Thinking HashMap is always "free"

A HashMap gives O(1) lookups but costs O(n) space. It's a time-space tradeoff.

💼 Interview Tips

  • Always state both time and space complexity when explaining your solution
  • Start with brute force, then optimize — show your thought process
  • Most competitive programming limits: ~10⁸ operations/second. So for n=10⁵, O(n²) = 10¹⁰ → too slow!
  • Quick guide for max n:
    • n ≤ 20 → O(2ⁿ) or O(n!) OK (backtracking)
    • n ≤ 3000 → O(n²) OK
    • n ≤ 10⁵ → need O(n log n) or O(n)
    • n ≤ 10⁸ → need O(n) or O(log n)

📝 Quick Revision

  • Big O = worst case upper bound. Drop constants and lower terms.
  • Nested loops multiply: O(n) × O(n) = O(n²)
  • Sequential code adds: O(n) + O(m) = O(n + m)
  • Recursion: time = branches^depth, space = depth
  • Always mention time AND space complexity

🃏 Flash Cards

What does O(log n) mean intuitively?
The problem size is cut in half at each step. For n=1,000,000 you only need ~20 steps. Example: Binary Search.
O(n² + 5n + 100) simplifies to?
O(n²). Drop constants and lower-order terms. As n grows, n² dominates.
What's the space complexity of recursive Fibonacci?
O(n) — the call stack goes n levels deep. Time is O(2ⁿ) due to branching, but space is just O(n) because stack frames are reused after returning.
