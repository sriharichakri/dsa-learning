Variables & Data Types

Phase 1 Beginner Friendly

📖 What Are Variables?

A variable is a named container that stores a value in your computer's memory. Think of it as a labeled box — the label is the variable name, and whatever you put inside the box is the value.

Every program needs to store and manipulate data. Variables give you a way to name that data so you can refer to it later — read it, update it, or pass it around.

💡 Key Insight

Variables are the atoms of programming. Every algorithm, every data structure, every line of code ultimately operates on variables. Understanding how they work in memory is the first step to understanding everything else in DSA.

🌍 Real-Life Analogy

📦 Labeled Boxes in a Warehouse

Imagine a warehouse full of boxes. Each box has a label (variable name), a size (data type), and contents (value). A small box labeled "age" holds the number 25. A bigger box labeled "name" holds the text "Alice". You can't fit a couch (huge data) in a shoe box (small type) — that's type mismatch!

  • Sticky note on a whiteboard: The note says "score" and the value written is 95. You can erase and write a new value anytime.
  • Hotel room key card: The card (variable) is linked to room 405 (memory address). Swipe it to access what's inside.
  • Contacts in your phone: "Mom" is the name (variable), the phone number is the value. You can update the number, but the name stays.

⚙️ Primitive Data Types

Primitive types store simple, single values directly in memory. They are the building blocks.

JavaScript
// Primitive types in JavaScript
let age = 25;           // Number (integer)
let price = 9.99;       // Number (float)
let isActive = true;    // Boolean

// Strongly typed languages require explicit types

boolean isActive = true;   // bool in C++
char grade = 'A';
String name = "Alice";     // std::string in C++

🔗 Reference Types

Reference types don't store the value directly. Instead, they store a reference (address) pointing to where the actual data lives in memory.

Primitive vs Reference in Memory
PRIMITIVE (value stored directly): Variable Memory ┌──────┐ ┌──────┐ │ age │──→│ 25 │ └──────┘ └──────┘ REFERENCE (address stored, points to object): Variable Memory Heap ┌──────┐ ┌──────┐ ┌──────────────────┐ │ arr │──→│ @4F2 │─────→│ [10, 20, 30, 40] │ └──────┘ └──────┘ └──────────────────┘
JavaScript
// Reference types: arrays, objects, functions
let arr = [1, 2, 3];           // Array
let person = { name: "Bob" };  // Object

// IMPORTANT: Copying a reference copies the ADDRESS, not the data!
let a = [1, 2, 3];
let b = a;       // b points to the SAME array
b.push(4);
console.log(a);  // [1, 2, 3, 4] ← a changed too!

// To make a real copy:
let c = [...a];  // Spread operator — new array
let d = a.slice(); // Another way to copy
⚡ The Copy Trap

With primitives, let b = a creates an independent copy. With reference types, it creates another pointer to the same data. This is the #1 source of bugs for beginners!

🧠 How Memory Works: Stack vs Heap

Your program uses two regions of memory:

┌──────────────────────────────────────────┐ │ MEMORY │ │ │ │ ┌─────────────┐ ┌─────────────────┐ │ │ │ STACK │ │ HEAP │ │ │ │ (fast, small)│ │ (large, slower) │ │ │ │ │ │ │ │ │ │ • Primitives │ │ • Objects │ │ │ │ • References │ │ • Arrays │ │ │ │ • Function │ │ • Strings* │ │ │ │ call frames│ │ • Dynamic data │ │ │ │ │ │ │ │ │ │ Auto-cleaned │ │ Garbage │ │ │ │ when function│ │ Collected │ │ │ │ returns │ │ │ │ │ └─────────────┘ └─────────────────┘ │ └──────────────────────────────────────────┘
  • Stack: Stores local variables and function calls. Fast (LIFO). Fixed-size. Automatically cleaned up.
  • Heap: Stores dynamically allocated objects. Larger but slower. Cleaned by garbage collector (JS, Java, Python) or manually (C/C++).

📝 Variable Declaration: var, let, const

In JavaScript, you have three ways to declare variables. Understanding the difference matters for interviews.

Featurevarletconst
ScopeFunctionBlockBlock
Re-assignableYesYesNo
Re-declarableYesNoNo
HoistedYes (undefined)TDZ*TDZ*

*TDZ = Temporal Dead Zone — accessing before declaration throws an error.

JavaScript
// var — function scoped, hoisted (avoid in modern JS)
var x = 10;
var x = 20; // No error! Re-declaration allowed 😬

// let — block scoped, not re-declarable
let y = 10;
// let y = 20; // ❌ SyntaxError

// const — block scoped, cannot be reassigned
const PI = 3.14159;
// PI = 3; // ❌ TypeError

// BUT const objects can be mutated!
const arr = [1, 2, 3];
arr.push(4);     // ✅ This works! The reference is constant, not the data.
// arr = [5, 6]; // ❌ Can't reassign the reference
🎯 Best Practice

Use const by default. Use let only when you need to reassign. Never use var.

🔄 Type Coercion & Conversion

JavaScript is dynamically typed — it automatically converts between types, which can cause surprises.

JavaScript
// Implicit coercion (automatic — be careful!)
"5" + 3    // "53"  (string concatenation!)
"5" - 3    // 2     (numeric subtraction)
true + 1   // 2     (true → 1)
"" == false // true  (both are "falsy")

// Explicit conversion (you control it)
Number("42")     // 42
String(42)       // "42"
parseInt("3.14") // 3
parseFloat("3.14") // 3.14
Boolean(0)       // false
Boolean("hello") // true
❌ The == vs === Trap
0 == ""     // true  (coercion!) ❌ Confusing
0 === ""    // false (strict, no coercion) ✅ Clear

null == undefined  // true
null === undefined // false

// Always use === in interviews and production code!

🔍 Dry Run: Variable Swapping

Let's trace through swapping two variables — a classic interview question.

JavaScript
let a = 5, b = 10;

// Method 1: Using a temporary variable
let temp = a;  // temp = 5
a = b;         // a = 10
b = temp;      // b = 5

// Method 2: Destructuring (modern JS)
[a, b] = [b, a];

// Method 3: Math trick (no extra variable)
a = a + b;  // a = 15
b = a - b;  // b = 5
a = a - b;  // a = 10
1
Initial state: a = 5, b = 10
a=5, b=10
2
temp = a → Save a's value in temp
a=5, b=10, temp=5
3
a = b → Copy b's value to a
a=10, b=10, temp=5
4
b = temp → Copy saved value to b
a=10, b=5, temp=5 ✅ Swapped!

💡 Why String Concatenation is O(n)

Strings are immutable. Concatenating two strings creates a brand new string and copies all characters. Concatenating in a loop can become O(n²) — use arrays and join() instead.

⚠️ Common Mistakes

❌ Confusing = (assignment) with == (comparison)
// ❌ BUG: This assigns, not compares!
if (x = 5) { ... } // Always true!

// ✅ CORRECT
if (x === 5) { ... }
❌ Using uninitialized variables
let total;
total += 10;  // NaN! Because undefined + 10 = NaN

// ✅ Always initialize
let total = 0;
total += 10;  // 10 ✅
❌ Integer overflow (in typed languages)
// In Java/C++:
int maxInt = 2147483647;  // Max 32-bit int
maxInt + 1;  // -2147483648! (overflow wraps around)

// Fix: use long (64-bit) for large numbers

💼 Interview Perspective

  • Know the difference between primitive vs reference types — interviewers test this with trick questions.
  • Understand pass by value vs pass by reference — primitives are copied, objects are shared.
  • Be able to explain stack vs heap memory in one sentence each.
  • Know the size limits of integer types (32-bit int max = 2³¹ - 1 ≈ 2.1 billion).
  • In JavaScript interviews, expect questions about hoisting, scope, and const vs let.

📝 Quick Revision Notes

  • Variable = named memory container. Has a name, type, and value.
  • Primitives (int, float, bool, char) store values directly on the stack.
  • Reference types (arrays, objects) store a pointer on the stack to data on the heap.
  • Copying primitives creates independent copies. Copying references shares the same data.
  • Use const by default, let when reassignment needed, never var.
  • Always use === (strict equality) in JavaScript.

🃏 Flash Cards

What is the difference between primitive and reference types?
Primitives store values directly (stack). Reference types store a memory address pointing to data on the heap. Copying a primitive copies the value; copying a reference copies the pointer (both point to same data).
What is the difference between stack and heap memory?
Stack: fast, fixed-size, stores local variables & function calls, auto-cleaned (LIFO). Heap: larger, slower, stores dynamic objects, cleaned by garbage collector.
Why should you avoid var in modern JavaScript?
var is function-scoped (not block-scoped), can be re-declared, and is hoisted as undefined. let and const are block-scoped and safer. Use const by default.
What happens when you do let b = a where a is an array?
b gets a copy of the reference (address), NOT a copy of the array. Both a and b point to the same array. Modifying one affects the other. Use [...a] or a.slice() for a real copy.
