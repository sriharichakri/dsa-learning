Segment Tree with Lazy Propagation

Phase 15 Hard

📖 What is Lazy Propagation?

Lazy Propagation is an optimization technique for Segment Trees that enables efficient range updates in O(log n) time, in addition to range queries. Without lazy propagation, updating every element in a range individually takes O(n log n). With it, pending updates are stored in a "lazy" array and only applied when needed.

The core idea is procrastination — don't do work until you absolutely must. When you update a range, mark the node as "pending" and move on. Only propagate the pending update downward when a child is actually accessed.

🏫 Real-Life Analogy: Lazy Teacher's Grade Book

Imagine a teacher needs to add 5 bonus marks to every student in sections A, B, and C. Instead of updating each student's score individually, the teacher writes "+5" on the section folder cover. Only when a specific student's score is needed (e.g., a parent inquires) does the teacher open that folder and apply the bonus. This "write it on the folder and deal with it later" approach is exactly lazy propagation!

⚙️ How Lazy Propagation Works

Range Update: Add 3 to indices [1, 4] on array [1, 3, 5, 7, 9, 11]
Before update: [36] lazy: [0] / \ [9] [27] lazy: [0, 0] / \ / \ [4] [5] [16] [11] lazy: [0, 0, 0, 0] / \ / \ [1] [3] [7] [9] Range update: add 3 to [1, 4] Step 1: Node [0,2] — partial overlap with [1,4] → Recurse children → Node [0,1] partial: Node [0,0] no overlap, Node [1,1] full → add 3 → Node [2,2] full overlap → add 3, set lazy Step 2: Node [3,5] — partial overlap with [1,4] → Node [3,4] full overlap → update sum += 3×2 = 6, set lazy[6] = 3 → Node [5,5] no overlap → skip After update: [48] lazy: [0] / \ [15] [33] lazy: [0, 0] / \ / \ [4] [8] [22] [11] lazy: [0, 0, 3, 0] / \ / \ ↑ pending! [1] [6] [7] [9] ← [3,4] children NOT updated yet
💡 Key Insight

Node [3,4] has sum=22 (correct: 10+12=22) but its children still show old values [7,9]. The lazy value 3 will be pushed down only when we query or update a child of that node.

🔧 Implementation

JavaScript — Segment Tree with Lazy Propagation
class LazySegmentTree {
    constructor(arr) {
        this.n = arr.length;
        this.tree = new Array(4 * this.n).fill(0);
        this.lazy = new Array(4 * this.n).fill(0);
        this.build(arr, 1, 0, this.n - 1);
    }

    build(arr, node, start, end) {
        if (start === end) {
            this.tree[node] = arr[start];
            return;
        }
        const mid = (start + end) >> 1;
        this.build(arr, 2 * node, start, mid);
        this.build(arr, 2 * node + 1, mid + 1, end);
        this.tree[node] = this.tree[2 * node] + this.tree[2 * node + 1];
    }

    // Push pending lazy updates to children
    pushDown(node, start, end) {
        if (this.lazy[node] !== 0) {
            const mid = (start + end) >> 1;
            // Apply to left child
            this.tree[2 * node] += this.lazy[node] * (mid - start + 1);
            this.lazy[2 * node] += this.lazy[node];
            // Apply to right child
            this.tree[2 * node + 1] += this.lazy[node] * (end - mid);
            this.lazy[2 * node + 1] += this.lazy[node];
            // Clear lazy for current node
            this.lazy[node] = 0;
        }
    }

    // Range Update: add val to all elements in [l, r]
    updateRange(l, r, val, node = 1, start = 0, end = this.n - 1) {
        if (r < start || end < l) return;           // No overlap
        if (l <= start && end <= r) {                // Total overlap
            this.tree[node] += val * (end - start + 1);
            this.lazy[node] += val;
            return;
        }
        this.pushDown(node, start, end);             // Partial overlap
        const mid = (start + end) >> 1;
        this.updateRange(l, r, val, 2 * node, start, mid);
        this.updateRange(l, r, val, 2 * node + 1, mid + 1, end);
        this.tree[node] = this.tree[2 * node] + this.tree[2 * node + 1];
    }

    // Range Query: sum of elements in [l, r]
    query(l, r, node = 1, start = 0, end = this.n - 1) {
        if (r < start || end < l) return 0;          // No overlap
        if (l <= start && end <= r) return this.tree[node]; // Total overlap
        this.pushDown(node, start, end);              // Partial overlap
        const mid = (start + end) >> 1;
        return this.query(l, r, 2 * node, start, mid) +
               this.query(l, r, 2 * node + 1, mid + 1, end);
    }
}

// Usage
const lst = new LazySegmentTree([1, 3, 5, 7, 9, 11]);
console.log(lst.query(1, 4));       // 3+5+7+9 = 24
lst.updateRange(1, 4, 3);           // Add 3 to indices 1..4
console.log(lst.query(1, 4));       // 6+8+10+12 = 36
console.log(lst.query(0, 5));       // 1+6+8+10+12+11 = 48

⚠️ Common Mistakes

❌ Forgetting to pushDown before recursing

In both updateRange and query, you must call pushDown before recursing into children on partial overlap. Otherwise, stale lazy values corrupt child nodes.

❌ Not scaling the lazy value by range length

When applying a lazy "add val" to a node covering [start, end], the sum increases by val × (end - start + 1), not just val. This is the most common bug.

❌ Mixing up assignment vs addition lazy

If your range update is "set all to val" (assignment), the lazy logic is different from "add val to all" (increment). Assignment lazy must override, not accumulate. Be clear which operation your tree supports.

💼 Interview Perspective

  • Lazy propagation is rarely asked in standard interviews but common in competitive programming and system design.
  • If an interviewer asks for efficient range updates + range queries, mention lazy propagation as the approach.
  • Understand the difference between point update (standard segment tree) and range update (needs lazy).
  • Can combine with other operations: range set, range add, range max — just change the merge logic and lazy application.
  • Alternative: BIT with range update trick works for range add + point query, but lazy segment tree is more general.

📝 Quick Revision Notes

  • Lazy propagation = "procrastinate updates until needed."
  • Extra lazy[] array stores pending updates for each node.
  • pushDown: apply lazy to children, clear parent's lazy.
  • Range update O(log n): on total overlap, update node + set lazy; on partial, pushDown then recurse.
  • Always pushDown before accessing children in both update and query.
  • For "add" lazy: sum += val × range_size, lazy accumulates. For "set" lazy: sum = val × range_size, lazy overrides.

🃏 Flash Cards

What problem does lazy propagation solve?
It enables range updates in O(log n) instead of O(n log n). Without it, updating every element in a range requires visiting each element individually. With lazy, we defer updates and only propagate when a node is actually accessed.
When must you call pushDown?
Before recursing into children during partial overlap in both updateRange and query. This ensures children have up-to-date values before we access them. On total overlap or no overlap, pushDown is not needed.
How does lazy "add" differ from lazy "set"?
Add: lazy values accumulate (lazy[node] += val). Set: lazy values override (lazy[node] = val, clear children's lazy). Set operations need a flag to distinguish "no pending" from "set to 0."
