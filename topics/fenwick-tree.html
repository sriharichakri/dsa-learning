Fenwick Tree (Binary Indexed Tree)

Phase 9 Hard

📖 What is a Fenwick Tree?

A Fenwick Tree (also called Binary Indexed Tree / BIT) is a data structure that supports prefix sum queries and point updates in O(log n) time. It uses a clever bit-manipulation trick to store partial sums efficiently.

Unlike a Segment Tree, a BIT is much simpler to implement (just two short functions!) and uses only O(n) space, but it's limited to operations that have an inverse (like sum, XOR) — it can't do range min/max directly.

💡 Key Insight

The magic of a Fenwick Tree is the lowest set bit (LSB). Each index i stores the sum of LSB(i) elements ending at i. To query prefix sum, you hop through indices by removing LSBs. To update, you hop by adding LSBs. Both paths have O(log n) hops.

🌍 Real-Life Analogy

🏢 Hierarchical Sales Reports

Imagine a company where each regional manager tracks sales for a few stores. To get total sales, you don't ask every single store — you ask a few managers who each cover a range of stores. If one store updates its numbers, only the managers covering that store need to update. The number of managers you need to contact is always small (log n).

  • Binary number system: The BIT leverages how binary numbers work — each bit position represents a power of 2.
  • Skip list: Like express lanes on a highway — some nodes cover larger ranges for faster traversal.

🔧 Fenwick Tree Implementation

JavaScript — Fenwick Tree (BIT)
class FenwickTree {
    constructor(n) {
        this.n = n;
        this.tree = new Array(n + 1).fill(0); // 1-indexed
    }

    // Point update: add val to index i — O(log n)
    update(i, val) {
        for (; i <= this.n; i += i & (-i)) {
            this.tree[i] += val;
        }
    }

    // Prefix sum query: sum of arr[1..i] — O(log n)
    query(i) {
        let sum = 0;
        for (; i > 0; i -= i & (-i)) {
            sum += this.tree[i];
        }
        return sum;
    }

    // Range sum query: sum of arr[l..r]
    rangeQuery(l, r) {
        return this.query(r) - this.query(l - 1);
    }
}

// Build from array
function buildBIT(arr) {
    const bit = new FenwickTree(arr.length);
    for (let i = 0; i < arr.length; i++) {
        bit.update(i + 1, arr[i]); // 1-indexed
    }
    return bit;
}

// Usage
const bit = buildBIT([1, 3, 5, 7, 9, 11]);
console.log(bit.query(4));        // prefix sum [1..4] = 1+3+5+7 = 16
console.log(bit.rangeQuery(2, 5)); // sum [2..5] = 3+5+7+9 = 24
bit.update(3, 5);                  // arr[3] += 5 (5 → 10)
console.log(bit.rangeQuery(2, 5)); // 3+10+7+9 = 29

⚠️ Common Mistakes

❌ Using 0-indexed arrays

Fenwick Trees are 1-indexed. Index 0 is a sentinel that stops the query loop. If you use 0-indexed input, add 1 to all indices when calling update/query.

❌ Confusing "set" vs "add" in update

The update function adds a delta. To set arr[i] to a new value, compute the difference: update(i, newVal - oldVal).

❌ Using BIT for range min/max

BIT only works for operations with an inverse (sum, XOR). Min/max don't have inverses, so you can't subtract to get range results. Use a Segment Tree instead.

💼 Interview Perspective

  • BIT is much simpler to code than Segment Tree — only two 3-line functions. Prefer it when possible.
  • The key formula: i & (-i) extracts the lowest set bit. Understand why it works (two's complement).
  • BIT can be used for counting inversions, order statistics, and coordinate compression problems.
  • Common interview problem: "Count of Smaller Numbers After Self" — use BIT with coordinate compression.
  • 2D BIT exists for matrix problems with updates and queries.

📝 Quick Revision Notes

  • Fenwick Tree (BIT): O(log n) point update + prefix sum. O(n) space. 1-indexed.
  • LSB formula: i & (-i). Query: subtract LSB. Update: add LSB.
  • Range sum = query(r) - query(l-1).
  • Only works for invertible operations (sum, XOR). Not for min/max.
  • Simpler than Segment Tree — use when range sum + point update is enough.

🃏 Flash Cards

How does a Fenwick Tree compute prefix sums in O(log n)?
Starting from index i, add BIT[i] to the sum, then subtract the lowest set bit: i -= i & (-i). Repeat until i = 0. Each step removes one bit, so at most log n steps.
What does i & (-i) compute?
It extracts the lowest set bit of i. For example: 6 (110) & -6 (010) = 2 (010). This determines how many elements BIT[i] covers: BIT[6] covers 2 elements.
Why can't BIT handle range minimum queries?
BIT relies on subtracting prefix results: rangeSum(l,r) = prefix(r) - prefix(l-1). Minimum has no inverse operation — you can't recover a range minimum by subtracting two prefix minimums. Use a Segment Tree instead.
