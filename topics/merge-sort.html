Merge Sort

Phase 4 Medium

📖 What is Merge Sort?

Merge Sort is a divide and conquer algorithm. It divides the array into two halves, recursively sorts each half, and then merges the two sorted halves into one sorted array.

It's guaranteed O(n log n) in all cases — best, average, and worst — making it one of the most reliable sorting algorithms.

💡 Key Insight

The "magic" of Merge Sort is in the merge step. Merging two sorted arrays into one sorted array is easy and takes O(n). The recursion ensures we always merge sorted subarrays.

🌍 Real-Life Analogy

📝 Sorting Exam Papers

Imagine a teacher has 100 unsorted exam papers. Instead of sorting all 100, she splits the pile in half and gives 50 to two assistants. Each assistant splits their pile again, and so on. When someone has just 1 paper, it's trivially "sorted." Then everyone merges their sorted piles back together by comparing the top papers from each pile. The final result: all 100 papers perfectly sorted!

  • Merging two sorted decks of cards: Take the smaller top card from either deck, repeat until done
  • Team sorting competition: Split work among teams, each team sorts their portion, then combine results
  • Library catalog merge: Two libraries merging their alphabetically sorted book lists into one

⚙️ How Merge Sort Works

Merge Sort: Splitting and Merging [38, 27, 43, 3, 9, 82, 10]
[38, 27, 43, 3, 9, 82, 10] ← Original / \ [38, 27, 43] [3, 9, 82, 10] ← Split / \ / \ [38] [27, 43] [3, 9] [82, 10] ← Split / \ / \ / \ [27] [43] [3] [9] [82] [10] ← Base cases [27] [43] [3] [9] [82] [10] \ / \ / \ / [27, 43] [3, 9] [10, 82] ← Merge \ / \ / [27, 38, 43] [3, 9, 10, 82] ← Merge \ / [3, 9, 10, 27, 38, 43, 82] ← Final Merge ✅
JavaScript — Merge Sort
function mergeSort(arr) {
    // Base case: array of 0 or 1 element is already sorted
    if (arr.length <= 1) return arr;
    
    // Divide: split array in half
    const mid = Math.floor(arr.length / 2);
    const left = mergeSort(arr.slice(0, mid));
    const right = mergeSort(arr.slice(mid));
    
    // Conquer: merge two sorted halves
    return merge(left, right);
}

function merge(left, right) {
    const result = [];
    let i = 0, j = 0;
    
    // Compare elements from both arrays, pick the smaller one
    while (i < left.length && j < right.length) {
        if (left[i] <= right[j]) {
            result.push(left[i]);
            i++;
        } else {
            result.push(right[j]);
            j++;
        }
    }
    
    // Append remaining elements
    while (i < left.length) result.push(left[i++]);
    while (j < right.length) result.push(right[j++]);
    
    return result;
}

🔗 The Merge Step — Heart of the Algorithm

Merging [3, 27, 38] and [9, 10, 82]
Left: [3, 27, 38] Right: [9, 10, 82] Result: [] ^i ^j Step 1: 3 < 9 → pick 3 Result: [3] Step 2: 27 > 9 → pick 9 Result: [3, 9] Step 3: 27 > 10 → pick 10 Result: [3, 9, 10] Step 4: 27 < 82 → pick 27 Result: [3, 9, 10, 27] Step 5: 38 < 82 → pick 38 Result: [3, 9, 10, 27, 38] Step 6: Right remaining → 82 Result: [3, 9, 10, 27, 38, 82] ✅

⏱️ Time & Space Complexity

CaseTimeWhy?
BestO(n log n)Always splits and merges, even if sorted
AverageO(n log n)log n levels of splitting × n work per level
WorstO(n log n)Same — divide and conquer guarantees this
SpaceO(n)Needs extra array for merging
💡 Why O(n log n)?

There are log n levels of recursive splitting (halving n until 1). At each level, the merge step processes all n elements. Total: n × log n.

⚠️ Common Mistakes

❌ Forgetting to Handle Remaining Elements

After the main merge loop, one array may still have elements. You must append all remaining elements from both arrays.

❌ Incorrect Mid Calculation
// ❌ For even-length arrays this can cause infinite recursion
let mid = Math.ceil(arr.length / 2); // May not reduce size

// ✅ Safe: always reduces problem size
let mid = Math.floor(arr.length / 2);
❌ Not Using Stable Comparison (<=)

Using < instead of <= in the merge makes it unstable. Use <= to preserve relative order of equal elements.

❌ Ignoring Space Complexity

Merge Sort needs O(n) extra space. If memory is tight, consider in-place algorithms like Quick Sort or Heap Sort.

💼 Interview Tips

  • Go-to for guaranteed O(n log n): Unlike Quick Sort, Merge Sort never degrades to O(n²)
  • Linked list sorting: Merge Sort is ideal for linked lists (no random access needed, O(1) extra space)
  • "Count inversions" problems: Modified Merge Sort can count inversions in O(n log n)
  • External sorting: Merge Sort is used for sorting data that doesn't fit in memory (database sorting)
  • Stable sort: When stability matters, Merge Sort is preferred over Quick Sort

🏗️ Where Merge Sort Is Used

  • Python's Timsort: Python's built-in sort is a hybrid of Merge Sort and Insertion Sort
  • Java's Arrays.sort(): Uses Merge Sort for object arrays (stability required)
  • External sorting: Databases use Merge Sort to sort data too large for memory
  • Git merge: The merge operation in version control follows a similar merge strategy

📝 Quick Revision Notes

  • Merge Sort = split in half → recursively sort → merge sorted halves
  • Always O(n log n) — best, average, and worst case
  • Space: O(n) — needs extra array for merging
  • Stable sort — equal elements keep original order
  • Ideal for linked lists, external sorting, and counting inversions

🃏 Flash Cards

What is the time complexity of Merge Sort in ALL cases?
O(n log n) — always. It always splits into log n levels and does n work per level. No worst-case degradation.
What is the space complexity of Merge Sort and why?
O(n) — because the merge step needs an auxiliary array to hold the merged result. This is the main drawback vs Quick Sort.
How can Merge Sort count inversions?
During the merge step, when an element from the right array is picked before remaining left elements, count += (remaining elements in left). Total count = number of inversions.
Why is Merge Sort preferred for linked lists?
Linked lists don't support random access. Merge Sort only needs sequential access. Also, merging linked lists requires O(1) extra space (just rewire pointers).
